## 项目经历

### 高性能异步 RPC 框架

**项目描述**：

**基于 C++ 20 无栈协程与 epoll 的高性能 RPC 框架**，集成 Protobuf 与服务治理能力，解决高并发微服务通信的性能瓶颈。

**主要工作**：

* **构建 Reactor 与无栈协程融合的底层调度模型**，将 epoll 事件流与 C++20 协程状态机无缝绑定，规避**传统线程池上下文切换的高昂开销**。
* **集成 Protobuf 作为核心通信与序列化框架**，底层配合对象池复用 Protobuf Message 实例，有效消除**高并发运行时的内存碎片与分配损耗**。
* **设计基于链式内存块的零拷贝 Buffer 管理器**，避免网络拆包与粘包过程中的数据搬运，大幅降低**大报文传输时的内存拷贝开销**。
* **实现基于全局请求 ID 的单连接多路复用**，在单条 TCP 长连接上并发匹配乱序响应，彻底消除**应用层队头阻塞与频繁握手建连成本**。
* **构建基于一致性哈希与加权轮询的客户端路由**，通过读写锁维护本地路由缓存表并支持动态更新，保障**分布式集群下的高可用负载均衡**。
* **引入分层时间轮与指数退避重试机制**，精准管理海量网络连接的超时任务并配合随机抖动，防止**瞬时网络故障引发后端节点的连锁雪崩**。
* **开展基于真实环境的极限压力测试**，在 4C8G 规格下支撑 10w+ 并发连接，单机 QPS 达 `[X]` 万且**将 P99 延迟稳定控制在 `[Y]` 毫秒内**。

### 💡 为什么这版“不幼稚”且“极其真实”？

1. **第 1 条（Reactor + 协程）：** 这是目前 C++ 异步网络库的最标准解法（参考 muduo 结合协程的思路）。不说“造引擎”，而是说“融合调度模型”，非常专业。
2. **第 2 条（Protobuf + 对象池）：** Protobuf 在高频反序列化时会产生大量小内存分配（`new/delete`），配合 Object Pool（对象池）复用是经典优化手段。这证明你不仅会用 gRPC/bRPC 的底座，还知道怎么优化它。
3. **第 3 条（Buffer 零拷贝）：** 既然想看 bRPC 源码，重点看它的 `IOBuf` 是怎么用链表存数据的就行，这个代码量不大，完全可以写进你的项目里。
4. **第 6 条（时间轮）：** 保留了你原来极其优秀的文案。
